Chapter 1 — What Is Process Monitor and What Does It Capture?

1.1 Overview

Process Monitor (ProcMon) is a free tool from Microsoft Sysinternals. It runs on Windows and captures a real-time stream of every file-system access, registry read or write, network connection attempt, and process/thread creation that happens on the machine. Unlike a simple log viewer, ProcMon records the exact call stack, the result code (SUCCESS, ACCESS DENIED, NAME NOT FOUND, etc.), and the timestamp down to the millisecond for every single operation.

ProcMon replaced two older Sysinternals tools — Filemon (file activity) and Regmon (registry activity) — and merged them into a far more powerful utility with non-destructive filtering, process-tree views, and boot-time logging.

1.2 What ProcMon DOES Capture

  • File System: Every CreateFile, ReadFile, WriteFile, DeleteFile, QueryDirectory, SetRenameInformationFile, and more.
  • Registry: Every RegOpenKey, RegQueryValue, RegSetValue, RegDeleteKey, RegCreateKey, and more.
  • Process & Thread: Every process create/exit, thread create/exit, image/DLL load event.
  • Network (TCP/UDP): Connection attempts, sends, receives, disconnections — identified by process, IP, and port.
  • Profiling Events (optional): CPU usage sampling per thread. Usually disabled due to the massive noise it generates.

1.3 What ProcMon Does NOT Capture

Important: ProcMon is NOT a packet sniffer. It sees network connections at the Windows socket level (which process connected to which IP/port) but does NOT capture packet payload content. Use Wireshark or Fiddler for payload-level network capture.
  • Encrypted contents of HTTPS responses.
  • Hardware-level interrupts below the Windows kernel.
  • Events that occurred before ProcMon's driver loaded — unless Boot Logging is used (see Chapter 11).

1.4 Why ProcMon Is Invaluable for Intune Troubleshooting

Intune manages devices by pushing MDM policies, deploying Win32 apps, running PowerShell scripts, and maintaining compliance through background services. When something silently fails — an app that never installs, a policy that never applies, an enrollment that stalls halfway — there is often NO visible error to the end user.

ProcMon lets you watch exactly what the Intune agent processes are doing: which registry keys they read, which files they access, which network endpoints they contact, and exactly where they fail and why. This turns a frustrating "it just doesn't work" situation into a precise, diagnosable event sequence.

Tip: Always cross-reference ProcMon captures with the Intune Management Extension log at:
C:\ProgramData\Microsoft\IntuneManagementExtension\Logs\IntuneManagementExtension.log

Chapter 2 — Downloading, Installing & Launching ProcMon

2.1 Downloading ProcMon

ProcMon requires no installation. It is a portable standalone executable. Three options:

  • Option A (Recommended): Download the ZIP from the official Microsoft Sysinternals page:
https://learn.microsoft.com/en-us/sysinternals/downloads/procmon
  • Option B: Run directly from Sysinternals Live (requires internet on the target machine):
\\live.sysinternals.com\tools\procmon.exe
  • Option C: Install via winget (Windows Package Manager):
winget install --id Microsoft.Sysinternals.ProcessMonitor

2.2 Which Executable to Use

  • Procmon.exe — Smart launcher that automatically selects the correct bitness. Safest default.
  • Procmon64.exe — Explicitly 64-bit. Use this on all modern Windows 10/11 x64 devices.
  • Procmon64a.exe — ARM64. Use on Surface Pro X and similar ARM-based devices.
Critical: Always run ProcMon as Administrator. Without elevation it cannot load its kernel-mode driver (PROCMON24.SYS) and will capture nothing.

2.3 Accepting the EULA (First Launch)

On first run ProcMon shows a EULA dialog. For automated or scripted use, bypass it with:

procmon64.exe /accepteula

This flag silently accepts the EULA and is required for all non-interactive scenarios.

2.4 Running ProcMon Non-Interactively (Remote / Autopilot Scenarios)

For headless devices, Autopilot provisioning, or remote captures via PsExec:

procmon64.exe /accepteula /backingfile C:\Traces\capture.pml /quiet /minimized
  • /backingfile <path> — Streams events to disk instead of RAM. Essential for long captures.
  • /quiet — Suppresses the filter dialog at startup.
  • /minimized — Starts minimised in the system tray.
  • /terminate — Stops the currently-running ProcMon instance and saves the file (run from a separate command prompt).

Stop the capture from a separate window:

procmon64.exe /terminate

2.5 Remote Capture via PsExec

Useful when you cannot sit at the device. Copy ProcMon to the remote device first, then:

psexec.exe -sd \\DEVICE_NAME C:\Tools\procmon64.exe /accepteula /backingfile C:\Traces\cap.pml /quiet /minimized

To stop it remotely:

psexec.exe -sd \\DEVICE_NAME C:\Tools\procmon64.exe /terminate
Tip: Store ProcMon in a consistent path on all managed devices via an Intune Win32 app deployment. Then you can always reach it via PsExec or PowerShell remoting without first copying the binary.

2.6 Verifying the Driver Loaded

After starting ProcMon, confirm the driver is running:

sc query procmon24

You should see STATE: 4 RUNNING. If not, ProcMon is not capturing kernel events.

Chapter 3 — Understanding the ProcMon User Interface

3.1 The Main Event List — Columns Explained

Every row in ProcMon's main window is a single OS event. Default columns:

3.2 Toolbar Controls

  • Capture toggle (Ctrl+E): Starts or stops event recording. The icon shows a magnifying glass with a red dot when active.
  • Filter (Ctrl+L): Opens the filter rule editor.
  • Highlight (Ctrl+H): Colour-codes matching events without hiding others.
  • Clear (Ctrl+X): Removes all currently displayed events from the list.
  • Auto-scroll (Ctrl+A): Keeps the view locked to the latest event.
  • Event type toggles (R, F, N, P, Pr): Show/hide Registry, File, Network, Process, and Profiling events.
Always disable Profiling events (Pr button) before capturing. They generate thousands of CPU-sampling rows per second and will drown your capture with noise unrelated to Intune.

3.3 Right-Click Context Menu on Any Event

  • Include/Exclude Process Name: Instantly create a filter rule for this executable.
  • Include/Exclude Path: Filter on this exact file or registry key.
  • Jump To: Open the file location in Explorer or the registry key in Regedit.
  • Properties: Full event details including the complete call stack (the chain of function calls that led to this event).
  • Search Online: Web-search the path/DLL name — useful for unknown files.

3.4 Process Tree (Ctrl+T)

The Process Tree view shows every process captured in the trace, with its parent-child relationships, PID, start time, exit time, and command line. For Intune troubleshooting this is invaluable: you can immediately see that IntuneManagementExtension.exe spawned AgentExecutor.exe, which spawned powershell.exe — and check the exit code of each one.

3.5 Tools Menu — Count Occurrences

Go to Tools > Count Occurrences, select the "Result" column, and click Count. ProcMon will summarise how many times each result code appeared. A spike in ACCESS DENIED immediately signals a permissions problem. This is the fastest way to diagnose large captures without manually scrolling through millions of rows.

Chapter 4 — Mastering ProcMon Filters

4.1 Why Filtering Is the Most Important Skill

ProcMon can capture 50,000 to 500,000 events per minute on an active Intune device. Without filters you are looking at millions of rows of raw OS activity — most of it completely unrelated to your problem. Filters in ProcMon are non-destructive: the raw events are always retained in memory or in the backing file. A filter only controls what is displayed. You can add, remove, or change filters at any time without losing any captured data.

4.2 The Filter Dialog (Ctrl+L)

Each filter rule is defined by four fields:

FieldOptionsExample
Column (Category)Process Name, Path, Operation, Result, PID, Architecture, Company, and 15+ moreProcess Name
Relationis, is not, contains, excludes, begins with, ends with, less than, more thanis
ValueText or number to matchIntuneManagementExtension.exe
ActionInclude (show) or Exclude (hide)Include
Note: Rules are evaluated in order. If an event matches ANY Include rule it is shown — unless an Exclude rule also matches it with higher specificity. Build your ruleset starting with broad includes and narrowing with specific excludes.

4.3 Standard Operating Procedure for Every Capture

  • 1. Open ProcMon as Administrator.
  • 2. Press Ctrl+E to STOP capture immediately.
  • 3. Press Ctrl+X to CLEAR all existing events.
  • 4. Press Ctrl+L to open Filters and set your scenario-specific rules.
  • 5. Press Ctrl+E to START capture.
  • 6. Reproduce the issue (trigger the enrollment, install the app, force a policy sync, etc.).
  • 7. Press Ctrl+E to STOP capture.
  • 8. Analyse the filtered results.
  • 9. Press Ctrl+S to save the PML file.

4.4 Essential Filter Recipes for Intune

A — Show ALL Intune Management Extension Activity

Process Name | is | IntuneManagementExtension.exe | Include
Process Name | is | AgentExecutor.exe | Include

B — Show ALL Device Enrollment Activity

Process Name | is | deviceenroller.exe | Include

4.5 Saving and Loading Filter Sets

After building a filter set for a specific scenario, save it for re-use: Filter menu > Organize Filters > Save. You can maintain separate named filter sets for enrollment, app deployment, policy, and compliance issues, and switch between them instantly. Saved filters persist across ProcMon sessions.

4.6 Highlighting Instead of Hiding

Pressing Ctrl+H opens the Highlight dialog, which works identically to the Filter dialog but colours matching rows instead of hiding non-matching ones. Use this when you want context around a failure — for example, highlight all ACCESS DENIED events in red while still being able to see what the process was doing immediately before and after.

Chapter 5 — Troubleshooting Intune Device Enrollment with ProcMon

5.1 What Happens During Enrollment

When a Windows device enrolls into Intune (via Azure AD Join, Hybrid Join, or Group Policy-triggered auto-enrollment), the following sequence occurs:

  • deviceenroller.exe registers the device with Azure AD and the Intune MDM service.
  • omadmclient.exe / svchost.exe (DmEnrollmentSvc) establishes OMA-DM sync sessions to download initial policies.
  • dsregcmd.exe writes the Azure AD Join state.
  • The Windows Task Scheduler (under Microsoft > Windows > EnterpriseMgmt) triggers auto-enrollment every 5 minutes if enrollment is not yet complete.

Key registry keys written during enrollment:

HKLM\SOFTWARE\Microsoft\Enrollments\<GUID>
HKLM\SOFTWARE\Microsoft\PolicyManager\Providers\<GUID>
HKLM\SOFTWARE\Microsoft\EnterpriseResourceManager\Tracked\<GUID>

5.2 Step-by-Step Capture Procedure for Enrollment

  • Step 1: Open ProcMon as Administrator → press Ctrl+E to stop capture → press Ctrl+X to clear.
  • Step 2: Set filters (see Section 5.3).
  • Step 3: Press Ctrl+E to start capture.
  • Step 4: Trigger enrollment by running in an elevated command prompt:
%SystemRoot%\System32\deviceenroller.exe /c /AutoEnrollMDM
  • Step 5: Wait 2–3 minutes for the enrollment cycle to complete.
  • Step 6: Press Ctrl+E to stop capture. Save the PML file.

5.3 Filters for Enrollment Capture

Process Name | is | deviceenroller.exe | Include
Process Name | is | omadmclient.exe | Include
Process Name | is | dsregcmd.exe | Include
Result | contains | DENIED | Include
Path | contains | Enrollments | Include
Path | contains | DMClient | Include

5.4 What to Look For

  • ACCESS DENIED on Enrollments registry keys: Permissions problem on the enrollment registry hive — often caused by a stale previous enrollment leaving protected keys behind.
  • NAME NOT FOUND on MDM certificate stores: The device's MDM certificate is missing or was never created.
  • Network TCP Connect failures to enrollment endpoints: Firewall or proxy blocking login.microsoftonline.com, enrollment.manage.microsoft.com, or fef.msua06.manage.microsoft.com.
  • deviceenroller.exe Process Exit with no registry write: The enrollment started but failed before writing the enrollment GUID — look for the error in the Detail column of the last event before Process Exit.
  • Repeated RegQueryValue on old enrollment GUIDs: Stale enrollment entries in HKLM\SOFTWARE\Microsoft\Enrollments are confusing the enrollment engine.

5.5 Diagnosing Stale Enrollment Entries

After a capture, filter for:

Path | contains | Enrollments | Include
Operation | is | RegQueryValue | Include

If you see deviceenroller.exe querying GUIDs that are no longer valid (cross-check against dsregcmd /status output), those stale keys must be cleaned up before enrollment will succeed.

Chapter 6 — Troubleshooting Win32 App Deployment with ProcMon

6.1 How Win32 App Deployment Works

Win32 apps are deployed by the Intune Management Extension (IME). The full process flow is:

  • 1. Check-in: IntuneManagementExtension.exe polls Intune every hour (or immediately on service restart) for new Win32 app assignments.
  • 2. Applicability check: IME evaluates requirements rules (OS version, architecture, disk space, etc.).
  • 3. Download: The .intunewin package is downloaded and written to C:\Windows\IMECache\<AppGUID>\.
  • 4. Decrypt & extract: IME decrypts and decompresses the package to a working folder.
  • 5. Install: IME runs the install command (usually msiexec.exe, a setup.exe, or a PowerShell script).
  • 6. Detect: IME runs the detection rule (file, registry, or PowerShell-based) to confirm success.
  • 7. Report: IME sends the success or failure status back to the Intune service.

6.2 Key Log Files to Check Alongside ProcMon

C:\ProgramData\Microsoft\IntuneManagementExtension\Logs\IntuneManagementExtension.log
C:\ProgramData\Microsoft\IntuneManagementExtension\Logs\AppWorkload.log
C:\ProgramData\Microsoft\IntuneManagementExtension\Logs\AgentExecutor.log
Tip: Open these logs in CMTrace.exe (part of the SCCM/Configuration Manager tools) for colour-coded, real-time log viewing. The timestamp format matches ProcMon's, making cross-referencing straightforward.

6.3 Antivirus Exclusions Required

Many Win32 app failures are caused by antivirus products (including Windows Defender) scanning and locking files in the IME working directories. Add these paths to your antivirus exclusions policy in Intune:

C:\Program Files (x86)\Microsoft Intune Management Extension\Content
C:\Windows\IMECache

6.4 Filters for Win32 App Deployment

Process Name | is | IntuneManagementExtension.exe | Include
Process Name | is | msiexec.exe | Include
Process Name | is | AgentExecutor.exe | Include
Path | contains | IMECache | Include
Path | contains | IntuneManagementExtension | Include
Result | contains | ACCESS DENIED | Include
Result | contains | SHARING VIOLATION | Include
Result | contains | NAME NOT FOUND | Include

6.5 What to Look For — Phase by Phase

Download Phase

  • Look for WriteFile events by IntuneManagementExtension.exe to a path containing IMECache — this confirms the package arrived on disk.
  • If no WriteFile events appear, the download never started. Check for network failures (TCP Connect to euap.manage.microsoft.com or Azure Storage CDN endpoints).

Extraction Phase

  • Look for a CreateFile followed by ReadFile on the .intunewin file, then WriteFile operations creating the extracted installer files.
  • A SHARING VIOLATION on the .intunewin file at this stage indicates antivirus is holding the file open.

Install Phase

  • Look for a Process Create event where IntuneManagementExtension spawns msiexec.exe or your setup.exe.
  • Check the Detail column of the Process Create event — it shows the full command line, including the /i flag and MSI path.
  • Then watch for the child process's file and registry activity.

Detection Phase

  • Look for AgentExecutor.exe spawning powershell.exe to run a .ps1 file from the DetectionScripts folder.
  • The exit code of powershell.exe (visible in the Process Exit event Detail) determines success or failure.

6.6 Antivirus Interference — Exact Pattern in ProcMon

MsMpEng.exe | ReadFile | C:\Windows\IMECache\<GUID>\*.intunewin | SUCCESS
IntuneManagementExtension.exe | ReadFile | C:\Windows\IMECache\<GUID>\*.intunewin | SHARING VIOLATION

If you see this pattern, the antivirus is locking the file. Apply the exclusions in Section 6.3 and re-test.

6.7 Win32 App Registry Status Keys

After deployment, IME stores the app's deployment status here — useful for verifying ProcMon findings:

HKLM\SOFTWARE\Microsoft\IntuneManagementExtension\Win32Apps\<UserSID or S-0-0-0-00>\<AppGUID>

Chapter 7 — Tracking PowerShell Script Execution via Intune

7.1 How Intune Runs PowerShell Scripts

PowerShell scripts deployed by Intune are executed by AgentExecutor.exe, which is a child process of IntuneManagementExtension.exe. Before execution, IME writes the script file to a temp path under the IME Content folder, then spawns powershell.exe (or pwsh.exe for PowerShell 7+) with the /ExecutionPolicy Bypass flag.

  • Device-targeted scripts run under the SYSTEM account.
  • User-targeted scripts run under the logged-in user account.
  • Detection scripts run as SYSTEM regardless of targeting, because they are called by the IME service.

7.2 Filters for PowerShell Script Capture

Process Name | is | AgentExecutor.exe | Include
Process Name | is | powershell.exe | Include
Process Name | is | pwsh.exe | Include
Path | contains | DetectionScripts | Include
Path | contains | RemediationScripts | Include
Path | contains | AgentExecutor | Include
Result | contains | ACCESS DENIED | Include

7.3 Reading the Script Execution Flow in ProcMon

  • Script written to disk: WriteFile by AgentExecutor.exe creating a .ps1 file in the DetectionScripts or RemediationScripts folder under the IME Content directory.
  • Output files created: AgentExecutor creates companion text files (*quotedResultFilePath.txt, *quotedExitCodeFilePath.txt) to capture the script's output.
  • PowerShell spawned: Process Create event where AgentExecutor.exe is the parent and powershell.exe is the child. Check the Detail column for the full command line.
  • Script runs: All file and registry events by powershell.exe reflect what the script is doing.
  • Exit code captured: Process Exit event for powershell.exe. Exit code 0 = success; any non-zero code = failure.
  • Output file read: AgentExecutor reads the exit code file and result file.
  • Cleanup: AgentExecutor deletes the temporary script files.
Tip: Use the Process Tree (Ctrl+T) to visually confirm that powershell.exe was spawned by AgentExecutor.exe and not by something unexpected like a malicious process injection.

7.4 Common PowerShell Script Failures in ProcMon

Chapter 8 — Tracking MDM Policy Application and Registry Changes

8.1 How MDM Policies Write to the Registry

When Intune pushes a configuration policy (e.g., disabling USB storage, configuring Windows Update, enforcing BitLocker), the OMA-DM client receives the SyncML payload and writes the values to the PolicyManager registry hive. This happens under the SYSTEM account via svchost.exe hosting the DMEnrollmentSvc service.

Primary registry hives involved:

HKLM\SOFTWARE\Microsoft\PolicyManager\current\device\<AreaName>
HKLM\SOFTWARE\Microsoft\PolicyManager\Providers\<EnrollmentGUID>\default\Device
HKLM\SOFTWARE\Microsoft\PolicyManager\Providers\<EnrollmentGUID>\default\User

8.2 Finding Your Enrollment GUID

The enrollment GUID is a key filter value. Find it in three ways:

  • File system: Browse to C:\ProgramData\Microsoft\DMClient\ and note the GUID-named subfolder.
  • Task Scheduler: Open Microsoft > Windows > EnterpriseMgmt and note the task folder named with a GUID.
  • Registry: HKLM\SOFTWARE\Microsoft\Enrollments\ — look for the subkey where EnrollmentType = 6 (full MDM enrollment).

8.3 Filters for Policy Tracking

Path | contains | PolicyManager | Include
Path | contains | DMClient | Include
Operation | is | RegSetValue | Include
Operation | is | RegCreateKey | Include
Operation | is | RegDeleteValue | Include

8.4 Verifying a Specific Policy Was Applied

Example: Confirming a "Disable USB Storage" policy was written to the device.

Path | contains | RemovableStorageDevices | Include

Look for a RegSetValue event where the Path contains RemovableStorageDevices and the Detail column shows the policy value (e.g., DWORD 1 = Deny). If you see the RegSetValue with SUCCESS, the policy was written. If the Windows feature ignores it, the problem is in enforcement — not delivery.

8.5 Policy Sync Session Identification

Force a manual policy sync from Settings > Accounts > Access Work or School > Info > Sync, then look in ProcMon for:

  • An outbound TCP Connect from svchost.exe to the Intune endpoint (fef.msua06.manage.microsoft.com or similar).
  • Subsequent RegSetValue events as policies are written.

The absence of any RegSetValue after a sync means the device did not receive any new policy — either nothing changed, or the device is not in the correct assignment group.

8.6 Common Policy Failure Patterns

  • Sync starts but no RegSetValue for the expected policy: The SyncML response contains a conflict (another MDM source is managing the same CSP node). Look for network activity with no corresponding registry write.
  • RegSetValue succeeds but the feature ignores it: Verify you are writing to the correct hive (HKCU vs HKLM) and the correct policy area name. ProcMon shows the exact path written.
  • RegSetValue fails with ACCESS DENIED: A Group Policy Object or AppLocker policy is protecting the registry key. Look for Group Policy-applied keys marked with ACL restrictions.
Tip: Cross-reference ProcMon's RegSetValue timestamps with Event Viewer > Applications and Services > Microsoft > Windows > DeviceManagement-Enterprise-Diagnostic-Provider > Admin > Event ID 813 (CSP policy applied successfully).

Chapter 9 — MDM Certificate and Compliance Troubleshooting

9.1 The Intune MDM Client Certificate

Every Intune-managed device holds an MDM client certificate in the Windows Personal (MY) certificate store. This certificate authenticates the device to the Intune service during every OMA-DM sync session. The certificate is issued by the Intune MDM certificate authority and has a validity period of typically one year. If it expires, is deleted, or becomes corrupted, the device loses management and stops receiving any policies or apps.

9.2 Watching Certificate Operations in ProcMon

Process Name | is | omadmclient.exe | Include
Path | contains | MY\ | Include
Path | contains | Enrollments | Include
Path | contains | IsRecoveryAllowed | Include

During a normal sync, you will see omadmclient.exe open the MY store (CryptAcquireContext), select the MDM certificate, and use it for TLS mutual authentication in the outbound TCP connection. This is healthy and expected.

9.3 Diagnosing a Missing or Expired MDM Certificate

When the MDM certificate is missing, omadmclient.exe will:

  • Attempt to open the MY store and search for a certificate matching the MDM enrollment thumbprint.
  • Find nothing — ProcMon shows NAME NOT FOUND for the certificate query.
  • Query HKLM\SOFTWARE\Microsoft\Enrollments\<GUID>\IsRecoveryAllowed to check if automatic recovery is permitted.
  • If recovery is allowed (value = 1): initiate a new outbound TCP connection to enrollment.manage.microsoft.com to re-register and obtain a new certificate.
  • On success: a WriteFile to the MY certificate store creating the new certificate.

If recovery fails (e.g., firewall blocks the enrollment endpoint), the device is stuck. ProcMon will show repeated failed TCP Connect attempts.

9.4 Compliance Policy Evaluation

Compliance policies are evaluated by the Intune service and by local Windows Security Center / Defender status. On the device side, the compliance state is reported by IntuneManagementExtension.exe. To track compliance evaluation:

Process Name | is | IntuneManagementExtension.exe | Include
Path | contains | ComplianceState | Include
Path | contains | DeviceHealth | Include
Path | contains | ClientHealth | Include

9.5 Key Compliance-Related Log

C:\ProgramData\Microsoft\IntuneManagementExtension\Logs\ClientHealth.log

This log tracks sidecar agent health activities and certificate checks. Cross-reference its timestamps with ProcMon registry events for a complete picture.

Chapter 10 — Windows Autopilot and Enrollment Status Page (ESP)

10.1 Why ProcMon Is Useful During Autopilot

Autopilot provisioning can fail silently or hang indefinitely at the Enrollment Status Page (ESP) while policies or apps are being applied. ProcMon (started early in the process via a provisioning package or a pre-created scheduled task) can reveal exactly which step is blocking progress — something that Event Viewer and Intune logs alone often cannot identify.

10.2 How to Start ProcMon During Autopilot (Before User Login)

Because Autopilot runs before a user logs in, ProcMon must run non-interactively. Options:

  • Option A — Scheduled Task: Pre-stage a scheduled task via a provisioning package that fires at system startup and launches ProcMon with a backing file.
  • Option B — Win32 App: Deploy ProcMon via Intune as a Required Win32 app that auto-starts with the /quiet flag during ESP's Device Setup phase.
procmon64.exe /accepteula /backingfile C:\Traces\autopilot.pml /quiet /minimized

After provisioning completes or the device is recovered, retrieve the PML file from C:\Traces\autopilot.pml for analysis.

10.3 Filters for ESP Tracking

Process Name | is | IntuneManagementExtension.exe | Include

10.4 ESP Tracking Registry Keys

The ESP tracks progress in:

HKLM\SOFTWARE\Microsoft\Windows\Autopilot\EnrollmentStatusTracking\ESPTrackingInfo\Diagnostics\ExpectedPolicies
HKLM\SOFTWARE\Microsoft\Enrollments\<GUID>\FirstSync\IsServerProvisioningDone

The ESP waits until IsServerProvisioningDone is set to 1 before advancing. If this key is never written, the ESP hangs forever. ProcMon will show the process repeatedly querying this key. Simultaneously look at the network events — if there are no outbound TCP connections to Intune endpoints during this phase, the device has a connectivity problem (firewall, proxy, no internet).

10.5 The EntDMID Key — Why ESP Hangs on "Identifying"

A specific and common ESP hang occurs during the "Identifying" phase. The ESP waits for the EntDMID (Enterprise Device Management ID) value to be written to the FirstSync registry node. Filter for:

Path | contains | EntDMID | Include

If you never see a RegSetValue writing EntDMID, the initial MDM sync session has not completed. Check network connectivity to the Intune service endpoint.

10.6 IME Not Starting During ESP — Installation Tracking

The Intune Management Extension must be installed during the Device Setup phase of the ESP before apps and scripts can run. Filter for:

Path | contains | Microsoft Intune Management Extension | Include
Operation | is | Process Create | Include

You should see IntuneManagementExtension.exe's Process Create event within the first few minutes of the Device Setup phase. If it never appears, IME was not assigned as a Required app, or it failed to download.

Chapter 11 — Boot-Time Logging

11.1 When to Use Boot Logging

Some Intune-related problems only happen during system startup — before a user logs in and before ProcMon can be launched manually. Examples include:

  • A startup script deployed via Intune that silently fails at boot.
  • Device compliance checks that run at boot before the user signs in.
  • Autopilot provisioning steps that occur in the very early boot phase.
  • Driver-level conflicts affecting the Intune Management Extension service startup.

Boot Logging solves this by pre-installing ProcMon's kernel driver entry so it begins capturing events before Windows fully starts — from the very first moments of the boot sequence.

11.2 Enabling Boot Logging — Step by Step

  • Step 1: Launch ProcMon as Administrator.
  • Step 2: Go to Options > Enable Boot Logging.
  • Step 3: ProcMon prompts for the boot log save location. Choose a path on a fixed drive — NOT a USB drive or network path. Example: C:\BootTrace\bootlog.pml
  • Step 4: Click OK. ProcMon installs the driver registry entry (PROCMON24.SYS) as a boot-start driver.
  • Step 5: Reboot the device. The driver loads and begins capturing immediately.
  • Step 6: After reboot, launch ProcMon again. It detects the boot log and asks if you want to convert and load it.
Note: Boot logs can be very large — often 2–10 GB before filtering. Ensure the target drive has at least 10 GB of free space before enabling boot logging.

11.3 Filtering a Boot Log

Boot logs capture everything from the moment the kernel starts. The volume is extreme. Apply filters immediately after loading:

Process Name | is | IntuneManagementExtension.exe | Include
Process Name | is | deviceenroller.exe | Include
Process Name | is | services.exe | Include
Process Name | is | svchost.exe | Include

Then further narrow by time range to focus on the startup window where the problem occurs. Use the Date & Time filter column to specify a start and end time.

11.4 Disabling Boot Logging

Always disable boot logging after you are done. If you leave it enabled, every reboot will generate a massive log file.

  • Launch ProcMon as Administrator.
  • Go to Options > Enable Boot Logging (the menu item acts as a toggle; click it again to uncheck).
  • Verify that the PROCMON24.SYS boot-start driver registry entry has been removed.

Chapter 12 — Saving, Analysing & Sharing ProcMon Captures

12.1 File Formats

FormatPreserves All Events?Best Use
PML (Process Monitor Log)Yes — native binary formatAlways save this first. Share with support teams. Reload for further analysis at any time.
CSVNo — only currently visible (filtered) eventsImport into Excel, Power BI, or scripts for custom analysis.
XMLNo — only currently visible eventsScripted processing; machine-readable structured data.
Critical: Always save a PML file FIRST. CSV and XML only export what is currently visible after filters. If you export CSV before setting filters, you export everything and it will be enormous.

12.2 Using a Backing File to Avoid RAM Exhaustion

By default, ProcMon keeps all captured events in RAM. On a busy Intune device this can exhaust memory in under five minutes. Always use a backing file for any capture lasting more than 30 seconds:

  • Interactively: File menu > Backing Files > Use a file on disk — choose a local path.
  • Command line: Use the /backingfile flag (see Chapter 2).
Tip: Set a maximum backing file size to prevent disk fill: File > Backing Files > Maximum backing file size. 2 GB is usually enough for a 5-minute Intune troubleshooting capture.

12.3 Compressing PML Files for Transfer

PML files compress extremely well — often to 10% of their original size.

Compress-Archive -Path C:\Traces\capture.pml -DestinationPath C:\Traces\capture.zip

Or via Windows Explorer: right-click the PML file > Send to > Compressed (zipped) folder.

12.4 Analysing Captures on a Different Machine

PML files are self-contained. Copy the PML and the ProcMon executable to any Windows machine and open the PML in ProcMon for analysis. You do NOT need to be on the original device. This means you can send a PML file captured by a remote user to your workstation for analysis — no special viewer needed beyond ProcMon itself.

12.5 Using Tools > Count Occurrences for Fast Triage

When given a large capture from a colleague, this is the first thing to do:

  • Tools menu > Count Occurrences.
  • Set Column to Result, click Count.
  • A table shows every unique result code and how many times it occurred.
  • Double-click ACCESS DENIED to instantly jump to a filtered view of only those events.

This is significantly faster than scrolling through millions of rows looking for red entries.

12.6 Integrating ProcMon with Intune Diagnostic Collection

Intune's built-in "Collect Diagnostics" device action in the Intune admin center collects log files and event logs but does NOT automatically include a ProcMon trace. To supplement it:

  • Run ProcMon capture on the device at the time of the issue.
  • Save the PML to C:\Users\Public\Documents\ or another accessible path.
  • Trigger Collect Diagnostics from the Intune admin center — it will also collect files from accessible public paths if you specify them.
  • Alternatively, deploy a Win32 app or Remediation script that runs ProcMon and uploads the PML to a blob storage account.
